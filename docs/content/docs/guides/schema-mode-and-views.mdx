---
title: Schema mode and views
description: Define ClickHouse-aware tables and query-backed views with the staged schema API.
---

Schema mode is Quarry's richer path for users who want more than a plain
TypeScript row interface.

It adds:

- runtime table metadata
- ClickHouse-aware column helpers
- query-backed views

## Define tables

```ts
import {
  Date,
  DateTime64,
  String,
  UInt32,
  createClickHouseDB,
  defineSchema,
  table,
  view,
} from "@oorestisime/quarry";

const schema = defineSchema({
  users: table.mergeTree({
    id: UInt32(),
    email: String(),
    signup_date: Date(),
  }),
  event_logs: table.replacingMergeTree({
    user_id: UInt32(),
    event_type: String(),
    created_at: DateTime64(3),
  }),
});
```

The table helpers are explicit about engine families. That matters for things
like `FINAL`.

For the full column helper matrix, see the
[schema column types reference](/docs/reference/schema-types).

### Table engine helpers

These helpers do not generate DDL. They attach engine metadata inside Quarry,
which today mainly affects whether `.final()` is allowed on that source.

| Helper | Engine metadata | `.final()` in Quarry | Typical use |
| --- | --- | --- | --- |
| `table({...})` | `Table` (engine-unspecified) | No | Schema typing and insert normalization without modeling a specific engine family. |
| `table.memory({...})` | `Memory` | No | In-memory scratch or test tables. |
| `table.mergeTree({...})` | `MergeTree` | No | Plain MergeTree table metadata. |
| `table.replacingMergeTree({...})` | `ReplacingMergeTree` | Yes | Deduplicated or last-write-wins style tables. |
| `table.summingMergeTree({...})` | `SummingMergeTree` | Yes | Pre-aggregated numeric rollups. |
| `table.aggregatingMergeTree({...})` | `AggregatingMergeTree` | Yes | Aggregate-state tables. |
| `table.collapsingMergeTree({...})` | `CollapsingMergeTree` | Yes | Sign-based collapsing rows. |
| `table.versionedCollapsingMergeTree({...})` | `VersionedCollapsingMergeTree` | Yes | Collapsing rows that also carry a version. |

<Callout type="info">
In schema mode, Quarry currently allows `.final()` only for the helpers marked
"Yes" above. `table(...)`, `table.memory(...)`, and `table.mergeTree(...)`
are treated as not `FINAL`-capable.
</Callout>

## Add query-backed views

Views are staged after the base tables so that the `db` inside the callback
already knows the base schema.

```ts
const schema = defineSchema({
  users: table.mergeTree({
    id: UInt32(),
    email: String(),
    signup_date: Date(),
  }),
  event_logs: table.replacingMergeTree({
    user_id: UInt32(),
    event_type: String(),
    created_at: DateTime64(3),
  }),
}).views((db) => ({
  signup_events: view.as(
    db
      .selectFrom("event_logs as e")
      .select("e.user_id", "e.created_at")
      .where("e.event_type", "=", "signup")
  ),

  daily_signups: view.as(
    db
      .selectFrom("users as u")
      .selectExpr((eb) => ["u.signup_date", eb.fn.count().as("total_users")])
      .groupBy("u.signup_date")
  ),
}));
```

Then pass that schema to `createClickHouseDB`:

```ts
const db = createClickHouseDB({ client, schema });
```

## Query from views

Once defined, query-backed views behave like normal selectable sources.

```ts
const rows = await db
  .selectFrom("daily_signups as d")
  .select("d.signup_date", "d.total_users")
  .orderBy("d.signup_date", "asc")
  .execute();
```

## Projected views expose only projected columns

This is one of the most important properties of schema-backed views.

If the view only selects:

- `signup_date`
- `total_users`

then source columns that were not projected are not visible through the view.

```ts
db.selectFrom("daily_signups as d").select("d.signup_date", "d.total_users");

// type error
db.selectFrom("daily_signups as d").select("d.email");
```

That keeps the view definition as the single source of truth for what is
selectable.

## Query builder features still apply inside views

Because views are defined with the same query builder, you can use things like:

- `db.table("...").final()`
- joins
- `selectExpr(...)`
- common `eb.fn.*` helpers
- grouping and aggregation

Example:

```ts
const schema = defineSchema({
  event_logs: table.replacingMergeTree({
    user_id: UInt32(),
    created_at: DateTime64(3),
  }),
}).views((db) => ({
  formatted_events: view.as(
    db
      .selectFrom(db.table("event_logs").final().as("e"))
      .selectExpr((eb) => [
        "e.user_id",
        eb.fn.toString("e.user_id").as("user_id_text"),
        eb.fn.formatDateTime("e.created_at", "%Y-%m-%d").as("created_date_text"),
      ])
  ),
}));
```

## Common errors

| Error message | What it means | How to fix it |
| --- | --- | --- |
| `Source 'X' is not insertable.` | You called `db.insertInto("X")` for a schema source that is selectable but not insertable, such as a query-backed view. | Insert into the underlying base table instead of the view. |
| `FINAL is not supported for source 'X'.` | You called `.final()` on a source Quarry does not mark as `FINAL`-capable. | Use `FINAL` only on sources declared with a `table.*` helper that Quarry marks as `FINAL`-capable, or remove `.final()`. |
| `Unable to infer selectable columns for view 'X'. Ensure the view query only selects expressions with known schema metadata.` | Quarry could not determine the view's output column metadata from the query passed to `view.as(...)`. | Prefer direct column refs, `selectAll(...)`, or aliased typed expressions such as `eb.fn.toString(...).as("...")`. Avoid building a view entirely from raw SQL expressions with no inferred output metadata. |

## Current limits

Schema-backed views are still early. Today Quarry supports a useful first pass
of output metadata inference, but not every possible ClickHouse expression is
covered yet.

The broad rule is:

- direct refs and `selectAll` work well
- common cast/date/string/aggregate helpers work well
- more unusual expressions may need more inference support later

Decimal schema helpers are also intentionally deferred for now. That only
affects schema declaration: Quarry does not yet export `Decimal*` column
helpers for `defineSchema(...)`.

You can still cast expressions to decimal values with
[`eb.fn.toDecimal64(...)`](/docs/reference/expressions#type-conversions) and
`eb.fn.toDecimal128(...)`. Those are query-time cast helpers, not schema
column types.

The reason is not the input side. ClickHouse accepts decimal strings on insert
just fine. The harder question is read semantics.

Under `JSONEachRow`, decimal columns currently come back through the driver as
JavaScript `number` values in the cases Quarry has validated. That is
convenient for moderate values, but it can lose precision for large decimals.

So a future decimal helper should probably wait until Quarry decides whether it
is willing to make schema-aware select rewrites for runtime honesty. For
example, a schema that wants decimal values as strings may need selected columns
to compile as `toString(amount) AS amount`, not just change the TypeScript type.
