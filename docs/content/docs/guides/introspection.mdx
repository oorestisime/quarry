---
title: Schema introspection
description: Bootstrap a Quarry schema module from an existing ClickHouse database with the experimental Quarry CLI.
---

This flow is currently **experimental** and lives in the separate
`@oorestisime/quarry-cli` package, not the published
`@oorestisime/quarry` runtime package.

Use it when you already have ClickHouse tables and want a first-pass
`defineSchema(...)` module instead of writing everything by hand.

## What it does

The introspection command:

- reads object metadata from `system.tables`
- converts supported table and view DDL into a Quarry schema module
- writes an autogenerated header with the safe generation options used
- reports how many objects were generated and how many were skipped

## Run it

```bash
npx @oorestisime/quarry-cli introspect \
  --url http://localhost:8123 \
  --database analytics \
  --out schema.ts
```

You can also provide connection settings through environment variables:

```bash
export CLICKHOUSE_URL=http://localhost:8123
export CLICKHOUSE_DATABASE=analytics
export CLICKHOUSE_USER=default
export CLICKHOUSE_PASSWORD=secret

npx @oorestisime/quarry-cli introspect --out schema.ts
```

The generated file starts with an autogenerated header and should be treated as
generated code.

## Useful options

Use `--tablesOnly` when you want to bootstrap table metadata first and deal with
views later.

```bash
npx @oorestisime/quarry-cli introspect --database analytics --tablesOnly --out schema.ts
```

Use include and exclude filters to limit the generated surface area.

```bash
npx @oorestisime/quarry-cli introspect \
  --database analytics \
  --includePattern '^public_' \
  --excludePattern '^public_tmp_' \
  --out schema.ts
```

## What gets skipped

The command skips unsupported objects instead of failing the whole run whenever
it can still produce a trusted schema.

Today that usually means:

- dictionaries and materialized views are excluded up front
- unsupported table clauses are rejected per object
- complex views outside the supported subset are skipped
- views whose source objects were not generated are skipped


## Current view subset

The current parser is intentionally narrow. It works best for simple views such
as:

- `SELECT * FROM ...`
- direct projected columns
- supported `eb.fn.*`-style function calls with aliases
- `FINAL`
- simple `WHERE` predicates
- `GROUP BY`

It is not trying to be a full ClickHouse SQL parser.

If your view uses joins, subqueries, complex expressions, or broader SQL
syntax, expect it to be skipped for now.

## After generation

The generated file is a starting point, not a lockstep mirror of your database.

Review the output for:

- naming you want to keep or normalize
- views you want to rewrite manually
- table metadata that should be edited for clarity
- skipped objects reported by the CLI

Once you are happy with the generated schema module, use it the same way as any
handwritten Quarry schema. See
[Schema mode and views](/docs/guides/schema-mode-and-views).
